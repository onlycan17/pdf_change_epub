# 2026-03-14 LLM 문서 구조·맥락 재정리 계획

## 배경
- 현재 텍스트 기반 PDF는 페이지/청크 단위로 텍스트를 추출한 뒤 경계만 보정하고 EPUB으로 렌더링합니다.
- 이 방식은 문장 끝과 다음 문장 시작이 어색하게 끊기거나, 문단이 잘못 나뉘고, 제목과 본문 구조가 흐트러지는 문제가 남습니다.
- 스캔 PDF 경로도 OCR/LLM 합성 결과를 바로 EPUB으로 넘기기 때문에, 전체 문서 관점에서 한 번 더 구조를 다듬는 단계가 없습니다.

## 목표
- 기본 변환 경로에 LLM 기반 "문서 흐름 재정리" 단계를 추가합니다.
- 문장 연결, 문단 경계, 제목/목록/강조 구조를 함께 검토해 EPUB 본문 품질을 높입니다.
- 텍스트 기반 PDF와 스캔 PDF 모두 같은 보정 서비스를 재사용합니다.
- LLM 실패 시 즉시 원문으로 폴백(실패 시 기본 동작으로 복귀)하여 변환 자체는 계속되게 유지합니다.

## 비목표
- 문서 의미를 요약하거나 새 내용을 추가하지 않습니다.
- EPUB 렌더러의 XHTML 구조를 전면 교체하지 않습니다.
- 번역 기능이나 OCR 인식 정확도 자체를 이번 변경에서 다루지 않습니다.

## 영향 범위
- `app/services/text_context_service.py`
  - 청크 경계 보정 외에 문서 전체 흐름 재정리 메서드 추가
  - 일반 텍스트/마크다운(스캔 합성 결과)별 프롬프트 분리
  - 긴 문서를 여러 구간으로 나눠 안전하게 처리하는 분할 로직 추가
- `app/services/conversion_orchestrator.py`
  - 텍스트 추출 결과와 스캔 합성 결과에 문서 흐름 재정리 단계 연결
  - LLM 사용 통계 집계가 새 단계까지 반영되도록 조정
- `tests/test_text_context_service.py`
  - 문서 재정리 분할/폴백/구조 보존 테스트 추가
- `tests/test_conversion_orchestrator.py`
  - EPUB 생성 전에 문서 재정리 결과가 실제 반영되는지 검증
- `docs/TEXT_CLEANUP.md`
  - 파이프라인 설명에 "문서 흐름 재정리" 단계 반영

## 설계 메모
- 청크 보정은 "찢어진 종이 조각의 가장자리만 맞추는 작업"에 가깝습니다.
- 이번에 넣는 문서 흐름 재정리는 "붙인 종이를 책처럼 읽히게 문단과 문장 흐름을 다시 다듬는 편집"에 가깝습니다.
- 마크다운 경로는 `# 제목`, `- 목록`, `[[VERSE]]`, `[[LINE:n]]`, `[[MATHIMG:...]]` 같은 구조 신호를 보존해야 합니다.
- 일반 텍스트 경로는 제목/본문/문단 분리를 살리고, 페이지 표식과 문맥 연결을 자연스럽게 정리해야 합니다.

## 검증 계획
1. 백엔드 포맷
- `make fmt`

2. 백엔드 린트/타입 검사
- `make lint`

3. 백엔드 테스트
- `make test`

4. 프론트엔드 린트
- `cd frontend && npm run lint`

5. 프론트엔드 타입 검사
- `cd frontend && npm run type-check`

6. 프론트엔드 테스트
- `cd frontend && npm run test`

7. 프론트엔드 빌드
- `cd frontend && npm run build`

8. 프론트엔드 포맷 검사
- `cd frontend && npx prettier --check .`

## 롤백 전략
- 오케스트레이터에서 문서 흐름 재정리 호출만 제거하면 기존 동작으로 바로 복귀할 수 있습니다.
- 새 메서드는 `TextContextCorrector` 내부에만 추가되므로 파일 단위 롤백도 쉽습니다.
