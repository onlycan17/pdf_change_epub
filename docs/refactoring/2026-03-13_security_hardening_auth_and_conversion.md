[배경]
- 인증 토큰이 프론트엔드 `localStorage`에 저장되어 XSS(악성 스크립트 주입) 발생 시 탈취 위험이 있었습니다.
- 여러 API 엔드포인트가 `X-API-Key`를 헤더에서 받기만 하고 실제 값 검증을 하지 않아 보호 장치가 비어 있었습니다.
- 변환 작업은 생성자 정보가 저장되지 않아, `conversion_id`를 아는 사용자가 다른 사용자의 상태 조회·다운로드·취소를 시도할 수 있었습니다.
- 운영자 이메일 `onlycan17@gmail.com`는 실제 운영 계정으로 유지해야 하지만, 프론트엔드에 노출되면 불필요한 공격 표적이 됩니다.

[목표]
- 브라우저 인증을 백엔드 쿠키 중심으로 관리해 프론트엔드의 토큰 노출을 줄입니다.
- 운영자 이메일은 백엔드에서만 관리하고, 프론트엔드는 `is_privileged` 응답만 사용하도록 정리합니다.
- 변환 작업에 소유자 정보를 연결해 조회·다운로드·취소·재시도 권한을 검증합니다.
- 운영 환경에서 기본 비밀키와 기본 API 키로 서버가 기동되지 않도록 차단합니다.

[비목표]
- 전체 인증 체계를 OAuth 전용 구조로 전면 교체하지 않습니다.
- 브라우저용 API 키 제거 또는 CSP/CSRF 전면 개편은 이번 변경 범위에 포함하지 않습니다.
- 기존 테스트 데이터에 존재하는 운영자 데모 계정 자체는 삭제하지 않습니다.

[2026-03-13 Playwright 실측 결과]
- 실제 브라우저에서 `/login`과 `/register` 진입 시 Google 로그인 버튼이 개발 환경 원본 미허용 상태에서 `403`과 반복 초기화 경고를 발생시켰습니다.
- `GoogleSignInButton`은 스크립트 로드 직후 `window.google.accounts.id` 준비가 끝나기 전에 실패 처리될 수 있어, 사용자에게 잘못된 초기화 실패 문구를 보여줄 수 있었습니다.
- 프론트 일부 API 유틸리티는 쿠키 인증으로 전환된 뒤에도 `localStorage`의 구형 토큰을 읽어 `Authorization` 헤더를 구성하고 있었습니다.
- 로그인 폼은 `type="email"` 입력만 허용하는데, 백엔드 데모 계정 `testuser`와 `premiumuser`는 이메일이 아닌 사용자명 기반이라 브라우저가 제출 자체를 차단했습니다.

[추가 조치]
- Google 로그인은 허용된 원본에서만 렌더링하고, 개발 환경 미설정 시에는 실패 대신 안내 문구로 비활성화합니다.
- Google 스크립트 준비 대기와 중복 초기화 방지를 적용해 불필요한 콘솔 경고와 거짓 실패 메시지를 줄입니다.
- 프론트 인증 요청은 `httpOnly` 쿠키 중심으로 단순화하고, 남아 있는 구형 `localStorage` 토큰은 정리 대상으로만 취급합니다.
- 로그인 식별자 입력은 이메일과 사용자명 모두 허용해 실제 백엔드 인증 규칙과 맞춥니다.
