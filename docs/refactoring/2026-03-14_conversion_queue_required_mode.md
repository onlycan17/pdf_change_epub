# 2026-03-14 변환 큐 필수 모드 도입 계획

## 배경

- 현재 변환 시작 로직은 Celery/Redis 연결에 실패하면 `AsyncQueueService`가 직접 실행 모드로 자동 전환됩니다.
- 직접 실행 모드는 겉으로는 비동기처럼 보여도, 실제 변환 파이프라인 안에 PDF 분석/OCR 같은 무거운 작업이 포함되어 있어 웹 서버 한 프로세스가 오래 붙잡힐 수 있습니다.
- 그 결과 여러 사용자가 동시에 변환을 시작하면, "한 주문이 끝날 때까지 다음 주문이 밀리는 식당"처럼 직렬 처리처럼 보일 수 있습니다.
- 운영 관점에서는 큐 미가동 상태를 조용히 숨기는 것보다, 변환 시작 단계에서 명확하게 실패시키는 편이 원인 파악과 복구에 유리합니다.

## 목표

- Celery/Redis가 준비되지 않은 상태에서는 새 변환 작업을 받지 않도록 합니다.
- API가 503(Service Unavailable, 지금은 처리할 준비가 안 된 상태)로 명확한 오류를 반환하도록 합니다.
- 필요 시 환경 변수로만 직접 실행 폴백을 다시 허용할 수 있게 남겨 둡니다.

## 비목표

- Celery/Redis를 자동으로 설치하거나 실행하지 않습니다.
- 변환 파이프라인 자체를 프로세스 풀로 재작성하지 않습니다.
- 다운로드/상태 조회 흐름 전체를 새 저장소 구조로 바꾸지 않습니다.

## 변경 범위

- `app/core/config.py`
  - 직접 실행 폴백 허용 여부 설정 추가
- `app/services/async_queue_service.py`
  - 큐 강제 모드 예외 추가
  - Celery 미연결 시 직접 실행 대신 예외 발생 분기 추가
- `app/api/v1/conversion.py`
  - 큐 미가동 예외를 503 응답으로 변환
- `tests/test_async_queue_service.py`
  - 큐 필수 모드 테스트 추가
- `tests/test_integration.py`
  - API가 503을 반환하는 통합 테스트로 기대값 변경
- `docs/DEPLOYMENT.md`, `.env.example`
  - 운영 설정과 실행 방법 문서 동기화

## 검증 계획

1. 단위 테스트
   - Celery 연결 실패 시 큐 필수 모드에서 예외가 발생하는지 확인
   - 환경 변수로 직접 실행 폴백을 허용하면 기존 동작이 유지되는지 확인
2. 통합 테스트
   - `/api/v1/conversion/start`가 큐 미가동 시 503을 반환하는지 확인
3. 자동 검사
   - `./.venv/bin/python -m pytest`
   - `./.venv/bin/python -m ruff check app tests`
   - `./.venv/bin/python -m mypy app tests`
   - `./.venv/bin/python -m black --check app tests`

## 영향 범위

- 기존처럼 Celery가 정상 동작하는 환경에서는 사용자 동작 변화가 없습니다.
- Celery/Redis가 꺼진 환경에서는 "조용한 직접 실행" 대신 명확한 장애 응답이 보입니다.
- API/DB 스키마 변경은 없습니다.

## 롤백 전략

- 환경 변수로 직접 실행 폴백 허용 값을 켜서 기존 방식으로 되돌릴 수 있게 유지합니다.
- 필요 시 이번 변경 커밋만 되돌리면 원래의 직접 실행 폴백으로 복귀할 수 있습니다.
