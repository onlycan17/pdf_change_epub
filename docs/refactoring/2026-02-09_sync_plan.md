# 2026-02-09 리팩토링/동기화 계획서

## 배경
- 프로젝트의 실제 구현 상태(백엔드 API/비동기 큐/테스트)와 문서 설명(초기화 상태, 일부 미구현)이 불일치합니다.
- 일부 보안 관련 값(API 키)이 코드에 하드코딩되어 운영 환경 전환 시 위험이 있습니다.
- 변환 재시도 로직에서 원본 PDF 바이트를 재사용하지 않아 실패 복구 품질이 낮아질 수 있습니다.
- 프론트엔드 린트 설정 오타로 자동 검사 파이프라인이 실패합니다.

## 목표
- 문서와 코드의 현재 상태를 일치시킵니다.
- 하드코딩된 보안 값을 설정 기반으로 전환합니다.
- 재시도 시 원본 PDF 데이터를 재사용하도록 보완합니다.
- 프론트엔드 린트 검사를 정상화합니다.

## 비목표
- 신규 기능 추가(예: 새로운 결제 기능, 새로운 OCR 엔진) 범위는 포함하지 않습니다.
- 대규모 아키텍처 전환(예: DB/큐 교체)은 수행하지 않습니다.

## 영향 범위
- 백엔드 설정/의존성 계층: API 키 검증 방식
- 스캔 PDF 처리 서비스: LLM API 키 주입 방식
- 비동기 큐 서비스: 재시도 입력 데이터 원천
- 프론트엔드 품질 도구 설정: ESLint extends 항목
- 문서: README, architecture 문서

## 검증 계획
1. 백엔드 자동 검사
- `python -m black --check app tests`
- `python -m ruff check app tests`
- `python -m mypy app tests`
- `python -m pytest`

2. 프론트엔드 자동 검사
- `npm run lint`
- `npm run type-check`
- `npm run build`

3. 문서 검증
- README 링크가 실제 파일을 가리키는지 확인
- 아키텍처 문서와 `docker-compose.yml` 구성 일치 여부 확인

## 롤백 전략
- 변경 파일을 기능 단위로 분리 유지합니다.
- 문제 발생 시 파일 단위로 이전 상태 복원합니다.
- 우선순위가 높은 운영 이슈가 발생하면 문서 변경만 유지하고 코드 변경은 부분 롤백합니다.
