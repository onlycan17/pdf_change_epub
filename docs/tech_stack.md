# 기술 스택 명세서

## 1. 개요
이 문서는 PDF → EPUB 변환기 프로젝트에서 사용할 기술 스택을 정의하고 각 기술의 선택 이유를 설명합니다. 특히 한국어 처리에 최적화되고 React 기반 웹 인터페이스를 제공하기 위한 기술들을 포함합니다.

## 2. 백엔드 기술 스택

### 2.1 언어 및 런타임
- **Python 3.12+**
  - 선택 이유:
    - 풍부한 PDF 처리 라이브러리 (PyPDF2, pdfminer.six)
    - 강력한 OCR 라이브러리 (pytesseract) 지원
    - EPUB 생성 라이브러리 (ebooklib) 사용 가능
    - 데이터 과학 및 머신러닝 라이브러리와의 호환성
    - 비교적 낮은 메모리 사용량과 높은 성능

### 2.2 웹 프레임워크
- **FastAPI**
  - 선택 이유:
    - 빠르고 현대적인 Python 웹 프레임워크
    - 자동 API 문서 생성 (Swagger, ReDoc)
    - 타입 힌트 기반으로 개발 생산성 향상
    - 비동기 처리 지원으로 높은 성능
    - Pydantic을 통한 데이터 유효성 검사

### 2.3 PDF 처리
- **PyPDF2/pdfminer.six**
  - 선택 이유:
    - PyPDF2: 텍스트 기반 PDF 처리에 최적화
    - pdfminer.six: 더 복잡한 PDF 구조 분석 가능
    - 두 라이브러리의 조합으로 다양한 PDF 처리 가능

### 2.4 OCR 처리
- **PaddleOCR (Baidu)**
  - 선택 이유:
    - 한국어 특화 모델로 높은 인식률 제공 (약 95% 이상)
    - 경량 모델(PP-OCRv3 기준 약 90MB)로 서버 환경 최적화
    - 오픈소스이며 무료로 사용 가능 (Apache 2.0 라이선스)
    - Text Detection과 Recognition을 통합 처리하여 간편한 사용
    - CPU 환경에서도 충분히 빠른 처리 속도 (100ms/텍스트 라인)
    - 80개 이상의 언어 지원으로 다국어 문서 처리 가능
    - SVTR 알고리즘 도입으로 정확한 텍스트 인식
    - 한국어 문서에 특화된 성능과 안정성 보장

### 2.5 EPUB 생성
- **ebooklib**
  - 선택 이유:
    - EPUB3 표준 준수
    - 메타데이터 처리와 목차 생성 기능
    - 이미지 처리와 CSS 스타일 적용 가능

### 2.6 데이터베이스 (향후 확장)
- **SQLite (개발용) / PostgreSQL (운영용)**
  - 선택 이유:
    - SQLite: 설정이 간단하고 개발에 적합
    - PostgreSQL: 확장성과 안정성이 뛰어나 운영에 적합
    - SQLAlchemy ORM을 통한 데이터베이스 추상화

## 3. 프론트엔드 기술 스택

### 3.1 언어 및 런타임
- **TypeScript**
  - 선택 이유:
    - 정적 타입 검사로 런타임 에러 감소
    - 코드 자동 완성과 IDE 지원 향상
    - 대규모 프로젝트 관리에 용이
    - JavaScript와의 호환성 유지

### 3.2 프레임워크
- **React 18+**
  - 선택 이유:
    - 컴포넌트 기반 아키텍처로 재사용성 향상
    - 가상 DOM으로 높은 렌더링 성능
    - 풍부한 생태계와 커뮤니티 지원
    - 사용자에게 친숙한 인터페이스 구현 가능

### 3.3 빌드 도구
- **Vite**
  - 선택 이유:
    - 빠른 개발 서버 시작과 핫 모듈 교체(HMR)
    - 최적화된 빌드 성능
    - TypeScript, JSX, CSS 등 다양한 기능 기본 지원
    - 설정이 간단하고 확장성 있음

### 3.4 상태 관리
- **React Context API + useReducer**
  - 선택 이유:
    - Redux와 같은 외부 라이브러리 없이도 충분한 기능 제공
    - React 내장 기능으로 학습 곡선이 낮음
    - 프로젝트 규모에 따라 Redux Toolkit으로 확장 가능

### 3.5 라우팅
- **React Router DOM v6+**
  - 선택 이유:
    - React와 완벽한 통합
    - 직관적인 API와 선언적 라우팅
    - 코드 스플리팅과 지연 로딩 지원

### 3.6 UI 컴포넌트 라이브러리
- **Lucide React**
  - 선택 이유:
    - 가벼우면서도 일관된 디자인의 아이콘 제공
    - React 컴포넌트로 쉽게 사용 가능
    - MIT 라이선스로 자유롭게 사용 가능

### 3.7 스타일링
- **CSS Modules 또는 Styled Components**
  - 선택 이유:
    - CSS Modules: 네이밍 충돌 방지와 스코프 격리
    - Styled Components: 컴포넌트 기반 스타일링과 동적 스타일링 가능
    - 프로젝트 요구사항에 따라 선택

## 4. 개발 및 테스트 도구

### 4.1 코드 품질 관리
- **ESLint (프론트엔드)**
  - 선택 이유:
    - JavaScript/TypeScript 코드의 문법 및 스타일 검사
    - 커스텀 규칙 설정 가능
    - IDE와의 통합으로 실시간 피드백 제공

- **ruff (백엔드)**
  - 선택 이유:
    - Python 코드의 빠른 린팅과 포매팅
    - 다양한 규칙 세트 지원
    - 높은 성능으로 개발 속도 향상

- **Prettier**
  - 선택 이유:
    - 코드 포매팅 자동화로 일관된 코드 스타일 유지
    - 다양한 언어 및 파일 형식 지원
    - ESLint와의 통합 가능

### 4.2 테스트 프레임워크
- **Jest (프론트엔드)**
  - 선택 이유:
    - React 컴포넌트 테스트에 최적화
    - 스냅샷 테스트로 UI 변경 감지
    - 풍부한 매처와 모의 기능 제공

- **Pytest (백엔드)**
  - 선택 이유:
    - Python 테스트의 표준으로 널리 사용
    - 간단한 문법과 직관적인 API
    - 다양한 플러그인과 확장 기능

### 4.3 타입 검사
- **mypy (백엔드)**
  - 선택 이유:
    - Python 코드의 타입 안전성 보장
    - 런타임 에러 감소
    - IDE와의 통합으로 개발 생산성 향상
